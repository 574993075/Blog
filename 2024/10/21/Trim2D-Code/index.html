scripts/run_dtu.py

代码部分

导入和初始化

  • os: 用于与操作系统进行交互。
  • ThreadPoolExecutor: 用于管理并行执行的任务，将多个任务分配到不同的 GPU 上运行。
  • subprocess: 用于执行外部命令行命令，比如训练和评估模型。
  • time: 用于在不同操作之间引入延时。
  • torch: PyTorch 库，用于获取 GPU 信息等。

场景和参数设置

  • scenes: 包含多个要处理的场景名称。
  • factors: 每个场景的缩放因子，用来调整场景的分辨率。
  • output_dir 和 tune_output_dir: 模型训练和调优的输出路径。
  • iteration: 设置最终的迭代次数，决定训练和评估的深度。
  • position_lr_init: 每个场景的学习率，针对特定场景如 scan63 提供了自定义学习率。
  • jobs: 场景和缩放因子的配对任务列表。

训练函数 train_scene

  • train_scene: 用于训练和评估每个场景的模型。接受三个参数：gpu（使用的 GPU 编号）、scene（当前处理的场景）、factor（缩放因子）。
  • cmds: 包含一系列用于训练、剪枝、调优、渲染和评估的命令。通过 subprocess.run 执行这些命令。
  • scan_id: 通过截取 scene 的名称获取扫描 ID。

工作线程函数 worker

  • worker: 负责在特定的 GPU 上处理给定场景的任务。调用 train_scene 函数完成整个训练、调优和评估流程。

任务调度函数 dispatch_jobs

  • dispatch_jobs: 用于调度任务，确保所有 GPU 被合理分配并执行任务。处理如下：
    • 获取当前可用的 GPU，并避免已保留或排除的 GPU。
    • 当有可用 GPU 时，将 jobs 中的任务分配给这些 GPU，调用 executor.submit 来并行执行任务。
    • 每次检查任务完成的状态，并释放已经完成的 GPU，允许它们再次接受新任务。
    • 通过 time.sleep(5) 控制循环的频率，防止无效快速循环。

主程序入口

  • 使用 ThreadPoolExecutor 来管理最大 8 个并行任务，调用 dispatch_jobs 来执行所有任务。

/train_from_scratch\run_dtu.py

代码部分

导入和初始设置

  • os: 用于与操作系统交互（如路径操作）。
  • ThreadPoolExecutor: 线程池，用于在多个 GPU 上并行化执行任务。
  • subprocess: 执行外部命令行程序，比如训练、渲染和评估任务。
  • time: 用于控制时间延迟，避免高频无效循环。
  • torch: 用于获取 GPU 的数量和信息。

场景和参数设置

  • scenes: 包含了多个需要处理的场景。
  • factors: 对应每个场景的缩放因子，调整输入图像的分辨率。
  • output_dir: 输出训练结果的文件夹。
  • iteration: 指定迭代次数，最终模型在 30000 次迭代后进行评估。
  • jobs: 将 scenes 和 factors 打包成任务。

训练函数 train_scene

  • train_scene: 该函数执行一个完整的场景训练和评估过程。

    • gpu: 使用的 GPU 编号。
    • scene: 当前处理的场景。
    • factor: 缩放因子，用于调整图像的分辨率。
    • cmds: 包含多个用于训练、渲染和评估的命令，逐一执行这些命令。

工作线程函数 worker

  • worker: 这是每个 GPU 上运行的工作函数，执行 train_scene 进行训练和评估。

任务调度器 dispatch_jobs

  • dispatch_jobs: 该函数调度所有 GPU 和任务，确保每个 GPU 都被分配合适的任务，并在任务完成后释放 GPU 供下一个任务使用。

    • jobs: 等待执行的任务列表。
    • executor: 线程池执行器，用于并行处理任务。

主程序入口

  • 创建一个最大 8 个并行任务的线程池，并将所有任务调度到可用 GPU 上

前两个代码的差异

训练脚本的差异

第一个代码

  • 使用了多个训练脚本，包括 train.py、cull_pcd.py 和 tune.py 等。
  • 训练步骤
    • 先进行基础的训练 (train.py)。
    • 使用 cull_pcd.py 对点云进行剪裁。
    • 使用 tune.py 进一步调优模型。
    • 之后执行渲染 (render.py)、评估网格和点云 (evaluate_single_scene.py)。

第二个代码

  • 只使用了一个训练脚本 train_TrimGS.py。
  • 训练步骤
    • 直接进行训练 (train_TrimGS.py)。
    • 然后进行渲染 (render.py) 和评估 (evaluate_single_scene.py)。

总结：

  • 第一个代码包含了一个额外的“点云剪裁”和“模型调优”步骤，相对复杂，处理更细致。
  • 第二个代码则省略了这些额外步骤，训练过程更加简化。

迭代次数和模型输出

第一个代码：

  • iteration 设置为 7000，即模型在 7000 次迭代后进行渲染和评估。
  • 调优阶段使用了 30000 次迭代点云剪裁后的结果。

第二个代码：

  • iteration 设置为 30000，直接在 30000 次迭代后进行渲染和评估。

总结：

  • 第一个代码的训练迭代数更少，但在模型调优后使用了 30000 次迭代的数据。
  • 第二个代码直接在 30000 次迭代后进行评估。

调优与优化的差异

第一个代码：

  • 使用了 tune.py 脚本，显式地调整模型的参数，并使用 position_lr_init 来设置不同场景的学习率。
  • 在调优过程中执行了“密度化间隔”和“贡献裁剪间隔”的优化步骤。

第二个代码：

  • 没有专门的调优步骤，所有优化都通过 train_TrimGS.py 完成。

总结：

  • 第一个代码有专门的调优阶段，提供了更细粒度的控制（如学习率调整）。
  • 第二个代码没有专门的调优步骤，整体流程更直接。

命令执行顺序和复杂性

第一个代码：

  • 执行了 5 个不同的命令，每个步骤分别是训练、剪裁点云、调优、渲染和评估，流程复杂。

第二个代码：

  • 只执行了 4 个命令，分别是训练、渲染和两个评估，省略了剪裁点云和调优的步骤。

总结：

  • 第一个代码的执行步骤更多、涉及的脚本更复杂，适合对模型做更细致的调整。
  • 第二个代码则简化了流程，更适合快速完成整个训练、渲染和评估过程。

使用的训练脚本

第一个代码：使用了 train.py、cull_pcd.py、tune.py、render.py、evaluate_single_scene.py 等多个不同的脚本。

第二个代码：使用了 train_TrimGS.py、render.py、evaluate_single_scene.py 等较少的脚本。

总结：

  • 第一个代码使用了更多脚本，功能模块化。
  • 第二个代码使用较少的脚本，流程简洁。
